## 第 1 题答案 — 选择题 — ★★☆

**正确答案：A**

**详细解析：**

**区别**：
- **断点（Breakpoint）**：当 PC 到达指定地址时触发 → 监测"执行了什么代码"
- **Watchpoint（数据断点/监视点）**：当指定内存地址被读/写时触发 → 监测"谁修改了我的变量"

**限制**：
- Cortex-M3/M4 通常只有 2-4 个硬件 Watchpoint（由 DWT 单元提供）
- 监测的地址必须对齐（4 字节对齐用于 4 字节变量）
- 监测范围有限（通常 4 字节）
- 不支持复杂的条件表达式

**典型场景**：
- "我的全局变量被谁偷偷改了？" → 对该变量设置写入 Watchpoint
- "为什么这个中断标志位无法清除？" → 对该寄存器地址设置写入 Watchpoint

GDB 命令：`(gdb) watch variable_name` / `(gdb) watch *0x20000100`

## 第 2 题答案 — 简答题 — ★★☆

**参考答案：**

**远程调试流程**：

```bash
# 1. 启动 OpenOCD（或 J-Link GDB Server）
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg

# 2. 启动 GDB，连接远程目标
arm-none-eabi-gdb firmware.elf
(gdb) target extended-remote localhost:3333
(gdb) monitor reset halt
(gdb) load

# 3. HardFault 自动中断
(gdb) break HardFault_Handler
(gdb) continue
# 当进入 HardFault 时，查看栈帧获取故障 PC
(gdb) x/8wx $sp     # 查看栈帧
# stacked_pc 在 offset 6 (根据 EXC_RETURN 判断 MSP/PSP)

# 4. 查看故障寄存器
(gdb) x/1wx 0xE000ED28  # CFSR
(gdb) x/1wx 0xE000ED34  # MMFAR
(gdb) x/1wx 0xE000ED38  # BFAR

# 5. 查看外设寄存器（如 GPIOA）
(gdb) x/10wx 0x40020000  # GPIOA 基址
# 或用 CMSIS SVD 文件加载外设定义
(gdb) monitor reg        # 通过 JTAG/SWD 直接读寄存器

# 6. 设置硬件断点（Flash 中，最多 4-8 个）
(gdb) hbreak main.c:150
(gdb) hbreak *0x08001000
# 如果硬件断点用完，使用软件断点（仅 RAM 中有效）
```

**常见问题**：
- 优化后代码中变量被优化掉 → 用 `volatile` 或降低优化等级调试
- SWO/SWO 输出调试信息无需断点
- `backtrace` 在 Thumb 模式下可能需要使用 `set arm force-mode thumb`

**详细解析：**

远程调试是嵌入式开发日常操作，面试考察工具熟练度。

## 第 3 题答案 — 编程题 — ★★☆

**参考实现：**

```c
#define LOG_BUF_SIZE 64

typedef struct {
    LogEntry entries[LOG_BUF_SIZE];
    volatile uint32_t write_idx;  // ISR 写入位置
    uint32_t read_idx;             // 主循环读取位置
} DebugLog;

// ISR-safe 写入（只修改 write_idx）
void log_write_isr(DebugLog *log, LogLevel lv, const char *fmt, ...) {
    uint32_t idx = log->write_idx;
    LogEntry *entry = &log->entries[idx];
    
    entry->tick = get_system_tick();  // ISR-safe (假设实现是安全的)
    entry->level = lv;
    
    // 简化版格式化（vsnprintf 可能非重入，但大多数嵌入式环境安全）
    va_list args;
    va_start(args, fmt);
    vsnprintf(entry->msg, sizeof(entry->msg), fmt, args);
    va_end(args);
    
    // 内存屏障确保消息写入完成后再更新索引
    __DMB();
    log->write_idx = (idx + 1) % LOG_BUF_SIZE;
}

// 主循环中调用，输出所有积压日志
void log_flush(DebugLog *log) {
    while (log->read_idx != log->write_idx) {
        LogEntry *entry = &log->entries[log->read_idx];
        printf("[%6lu] %c: %s\n",
               entry->tick,
               "EWID"[entry->level],
               entry->msg);
        log->read_idx = (log->read_idx + 1) % LOG_BUF_SIZE;
    }
}

// 获取日志数量（诊断用）
uint32_t log_count(DebugLog *log) {
    int32_t diff = (int32_t)(log->write_idx - log->read_idx);
    if (diff < 0) diff += LOG_BUF_SIZE;
    return (uint32_t)diff;
}

// 使用示例
static DebugLog g_log;
// ISR 中:
// log_write_isr(&g_log, LOG_WARN, "ADC overrange: %d", val);
// main() 中:
// log_flush(&g_log);
```

**详细解析：**

日志系统设计要点：
1. **ISR 安全**：ISR 只写 write_idx，主循环只读 read_idx — 无锁设计
2. **环形缓冲**：满时覆盖旧数据，不会阻塞
3. **vsnprintf**：大多数嵌入式 libc（newlib-nano）的实现是线程安全的（但有缓冲区大小限制）
4. **内存屏障**：确保数据写入在索引更新前完成
5. **可扩展**：可通过 UART/SPI/SWO 输出，或保留在 RAM 供调试器查看

## 第 4 题答案 — 简答题 — ★★☆

**参考答案：**

1. **函数内 static 局部变量**：变量的生命周期变为整个程序运行期间（存储在 .data 或 .bss 段），但作用域仍限于函数内。常用于需要跨函数调用保持状态的场景。
2. **文件作用域 static 全局变量**：限制变量的链接属性为 internal linkage，即仅在当前 .c 文件内可见。其他文件无法通过 `extern` 引用。
3. **static 函数**：限制函数的链接属性为 internal linkage，仅在当前 .c 文件内可调用。

嵌入式系统中合理使用 static 的重要性：减少全局命名空间污染、防止其他模块意外修改数据、编译器可以更好地优化（内联等），有助于模块化和封装。

**详细解析：**

理解 static 的三种用法是嵌入式 C 面试中的高频问题。在资源受限的嵌入式系统中，正确使用 static 可以减少 RAM 使用并提高代码安全性。
