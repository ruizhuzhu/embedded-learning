## 第 1 题答案 — 编程题 — ★★☆

**参考实现：**

```c
typedef struct {
    uint8_t *buffer;
    uint32_t size;
    volatile uint32_t head;
    volatile uint32_t tail;
} RingBuf;

// 满判断：head 的下一个位置是 tail（牺牲一个存储单元）
static inline bool rb_is_full(RingBuf *rb) {
    return ((rb->head + 1) % rb->size) == rb->tail;
}

// 空判断：head == tail
static inline bool rb_is_empty(RingBuf *rb) {
    return rb->head == rb->tail;
}

bool rb_push(RingBuf *rb, uint8_t data) {
    uint32_t next = (rb->head + 1) % rb->size;
    if (next == rb->tail) {
        return false;  // 满
    }
    rb->buffer[rb->head] = data;
    // 内存屏障确保写入在 head 更新之前完成
    __DMB();
    rb->head = next;
    return true;
}

bool rb_pop(RingBuf *rb, uint8_t *data) {
    if (rb->tail == rb->head) {
        return false;  // 空
    }
    *data = rb->buffer[rb->tail];
    __DMB();
    rb->tail = (rb->tail + 1) % rb->size;
    return true;
}

// 获取已存储的数据数量
uint32_t rb_count(RingBuf *rb) {
    return (rb->head - rb->tail + rb->size) % rb->size;
}
```

**详细解析：**

**设计要点**：
1. **牺牲一个单元区分满/空**：满 = head+1 == tail（N-1 个有效数据），空 = head == tail。这是最优雅的方案，避免使用 count 变量的原子操作需求
2. **SPSC 无锁**：生产者只写 head，消费者只写 tail，各写各的，无竞争
3. **内存屏障**：`__DMB()` 确保数据写入在指针更新前完成，防止消费者读到不完整数据
4. **测试技巧**：head/tail 用 2 的幂次方 size → 位运算取模（`head & (size-1)`），更快

## 第 2 题答案 — 选择题 — ★★☆

**正确答案：A**

**详细解析：**

**CRC 系列对比**：

| 类型 | 校验值位宽 | 典型多项式 | 检错能力 | 典型应用 |
|------|-----------|-----------|---------|---------|
| CRC-8 | 8 位 | x⁸+x²+x+1 | 99.6% | 1-Wire, SMBus PEC |
| CRC-16 | 16 位 | x¹⁶+x¹⁵+x²+1 (CRC-16-IBM) | 99.998% | Modbus, USB |
| CRC-32 | 32 位 | x³²+x²⁶+...+1 | 99.99999998% | Ethernet, PNG, GZIP |

**Modbus RTU 使用 CRC-16-IBM**，多项式 0x8005，初始值 0xFFFF，结果异或 0x0000，先传低字节。

**优化技巧**：查表法比逐位计算快 8 倍以上。
```c
// 查表法 CRC-8
uint8_t crc8_table[256];  // 预计算
uint8_t crc8(uint8_t *data, uint32_t len) {
    uint8_t crc = 0x00;
    while (len--) crc = crc8_table[crc ^ *data++];
    return crc;
}
```

## 第 3 题答案 — 简答题 — ★★★

**参考答案：**

**FPU 上下文处理**：
- Cortex-M4F 有 32 个单精度浮点寄存器 S0-S31（每个 4 字节），加上 FPSCR 状态寄存器
- 如果每次上下文切换都保存/恢复所有 FPU 寄存器 → 额外 132 字节的栈空间和时间开销
- 大多数 RTOS 任务不使用浮点运算，全保存浪费严重

**惰性压栈（Lazy Stacking）**：
1. 进入异常/中断时，CPU 只自动压栈基本寄存器（R0-R3、R12、LR、PC、xPSR），不压 FPU 寄存器
2. CPU 设置 FPCA（Floating Point Context Active）位和 EXC_RETURN 的 bit 4
3. 如果新任务使用了 FPU 指令，CPU 检测到非活跃 FPU 上下文的访问 → 触发 UsageFault → 硬件自动将上一个任务的 FPU 寄存器压栈
4. 如果新任务根本不碰 FPU，永远不需要保存/恢复 FPU 上下文

**FreeRTOS 的实现**：
- 配置 `configUSE_TASK_FPU_SUPPORT = 2` 使用惰性压栈
- 任务创建时额外分配 FPU 上下文空间，但只在任务确实使用 FPU 时才使用
- 这大幅减少了上下文切换的 CPU 周期数

**详细解析：**

考察对硬件特性细节的深入理解，有 FPU 支持后如何高效做上下文切换。

## 第 4 题答案 — 选择题 — ★☆☆

**正确答案：B**

**详细解析：**

`volatile` 告诉编译器该变量的值可能在程序流程之外被改变（如硬件寄存器、ISR、其他线程），因此每次访问都必须从内存实际读取，不能使用缓存在寄存器中的值。这在访问内存映射 I/O 寄存器时是必需的。选项 C 是 `const` 的作用；选项 D 是 `extern` 的用途。
